# Wizard — Manual

## Что это

Wizard (визард) — инструмент для создания агентских систем в проектах. Запускается один раз на новый проект через `/bootstrap`, через диалог с senior/architect создаёт набор промптов-инструкций для агентов.

## Ключевое различие: Визард vs Агентская система

**Визард** — конфигуратор. Запускается один раз, задаёт вопросы, генерирует файлы. Работает в контексте Claude Code.

**Агентская система** — результат работы визарда. Набор промптов/скиллов, по которым агенты потом работают над задачами. Живёт в `.claude/commands/` целевого проекта.

Эти два понятия легко спутать, потому что оба используют субагентов, оба работают с кодовой базой. При любом обсуждении явно указывать, о чём речь: (визард) или (агентская система).

## Откуда идея

Подход из видео по контекстной инженерии. Ключевой принцип: код пишется только на последнем этапе, всё остальное — подготовка контекста. Каждый этап создаёт контекст для следующего.

### 4 этапа агентской системы (из видео):

1. **Research** — сужение кодовой базы до релевантной информации. Только факты, без мнений. Субагенты (агентской системы) сканируют код по направлениям, лид (агентской системы) синтезирует результат.
2. **Design** — архитектурное решение. C4 диаграммы, data flow, sequence diagrams (Mermaid). ADR для сложных решений. Стратегия тестирования. Ревьюируется человеком.
3. **Plan** — план реализации, разбитый на фазы. Каждая фаза — завершённая единица (можно закоммитить, протестировать). Не один гигантский файл, а несколько фаз.
4. **Implement** — написание кода по плану. Агенты с разделением ответственности: кодер, ревьюер качества, ревьюер безопасности, ревьюер соответствия дизайну. Quality gates между фазами: build, tests, linters, security, соответствие дизайну.

### Принципы из видео:

- Промпты должны быть специализированы под тип задачи (фича/баг) и стек (бэкенд/фронтенд). Универсальный промпт = плохое качество.
- Формула качества: корректность + полнота, при минимальном размере контекста и минимальном шуме.
- Research только описывает as-is, без советов и мнений.
- Design ревьюируется человеком — это gate контроля качества.
- Plan ревьюируется человеком — второй gate.
- При имплементации каждая фаза проходит все quality gates. Не прошла — не двигаемся дальше.
- Ревьюер никогда не правит код. Кодер никогда не ревьюирует. Разделение ответственности.

## Как работает визард

### Фаза 0 (визард): Понимание проекта

1. Субагент (визарда) сканирует документацию проекта (README, docs/, etc.)
2. Параллельно визард спрашивает человека: "О чём проект? Общими словами."
3. Субагент (визарда) сканирует код: стек, зависимости, структура, внешние сервисы, API, тесты, CI/CD
4. Визард сопоставляет доки и код. Если в доках есть что-то, чего не нашёл анализатор кода — запускает субагента (визарда) повторно с уточнением
5. Визард синтезирует понимание, задаёт уточняющие вопросы если есть расхождения или неясности

### Фаза 1 (визард): Промпт для Research-этапа (агентской системы)

Визард задаёт вопросы, специфичные для research-этапа:
- Какие области кода наиболее критичны?
- Есть ли неочевидные связи между модулями?
- Какие внешние системы взаимодействуют с проектом?
- etc.

Валидация: ответы проверяются через субагента (визарда) по коду. Если ответ не подтверждается кодом — уточняющий цикл.

Результат: промпт для research-агента (агентской системы) → сохраняется в файл.

### Фаза 2 (визард): Промпт для Design-этапа (агентской системы)

Вопросы про архитектурные принципы, стандарты, паттерны проектирования, подход к диаграммам.

Результат: промпт для design-агента → файл.

### Фаза 3 (визард): Промпт для Plan-этапа (агентской системы)

Вопросы про подход к разбивке на фазы, гранулярность, критерии завершённости фазы.

Результат: промпт для plan-агента → файл.

### Фаза 4 (визард): Промпт для Implement-этапа (агентской системы)

Вопросы про стандарты кодирования, линтеры, команды для build/test/lint, CI/CD pipeline.

Результат: промпт для implement-агента → файл.

### Фаза 5 (визард): Промпты для Review-этапов (агентской системы)

Вопросы про стандарты ревью, security-требования, compliance.

Результат: промпты для review-агентов (качество, безопасность, соответствие дизайну) → файлы.

## Принцип "от простого к сложному"

Первая версия визарда:
- Всё в одном контексте (без сброса между фазами)
- Субагенты (визарда) только для сканирования кода/доков и валидации ответов
- Только фичи (не баги)
- Без delta update

Улучшения — в TODO.md.

## Выход визарда

Набор файлов в `.claude/commands/` целевого проекта — локальные скиллы:
- research.md
- design.md
- plan.md
- implement.md
- review-quality.md
- review-security.md
- review-design.md
- quality-gates.md (чеклист проверок между фазами)
